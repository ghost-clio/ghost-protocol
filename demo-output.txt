  📜 ERC8004ENSBridge.sol — On-chain agent directory

     ┌─────────────┐     ┌──────────────────┐     ┌──────────────┐
     │  ENS (L1)   │────▶│ ERC8004ENSBridge │◀────│ ERC-8004 (L2)│
     │ ghost.eth   │     │   Links names    │     │ participantId│
     │ human name  │     │   to identities  │     │ agent scope  │
     └─────────────┘     └──────────────────┘     └──────────────┘

     Registration flow:
     1. Agent registers ERC-8004 identity on Base (L2)
     2. Agent calls registerIdentity() on bridge (L1) with their ENS name
     3. Bridge stores: ENS name ↔ participantId ↔ L2 chain ↔ manifest
     4. Other agents call resolveAgent() or lookupByParticipantId()
     5. Off-chain verification: check L2 tx receipt independently

     Two paths to discovery:
     A) Know a name?  → resolveAgent(namehash) → get participantId + manifest
     B) Found an ID?  → lookupByParticipantId() → get ENS name + manifest

     No oracle needed. No cross-chain messaging.
     The bridge stores attestations. Verifiers check L2 independently.

  🤝 Agent-to-agent trust scenario:

     Ghost Protocol wants to interact with treasury.eth...
     1. bridge.resolve("treasury.eth") → gets ERC-8004 identity
     2. bridge.assessTrust("treasury.eth") → HIGH (score 85/100)
        ✅ ENS resolves to valid address
        ✅ ERC-8004 agent identity found
        ✅ Identity verified via bridge contract
        ✅ Agent manifest available
        ✅ Capabilities: treasury-management, defi-execution
     3. Ghost Protocol proceeds with transaction
     4. If trust < threshold → refuse to interact

     This is how agents build trust without humans in the loop.
